# 1. Executive summary

**Рабочее название:** лучше использовать короткое имя бинарника и более полное имя продукта.

Хорошие варианты:

| Product name      |   CLI binary | Комментарий                                                 |
| ----------------- | -----------: | ----------------------------------------------------------- |
| **DeepSeek Code** |        `dsc` | Просто, ясно, но может конфликтовать по бренду.             |
| **DSX Code**      |        `dsx` | Лучшее имя для dev tool: коротко, технично, легко печатать. |
| **SeekCode**      |       `seek` | Хорошо звучит, но менее явно связано с DeepSeek.            |
| **DeepDev**       |    `deepdev` | Универсально, но менее уникально.                           |
| **Seek Agent**    | `seek-agent` | Описательно, но длиннее.                                    |

Моя рекомендация: **DSX Code**, бинарник **`dsx`**. В документации можно писать: “DSX Code — DeepSeek-powered coding agent”.

Ключевой вывод: **не строить wrapper вокруг DeepSeek API**. Нужно строить **локальный agent runtime**:

```text
DeepSeek V4 = model provider
DSX Code    = permissions + tools + memory + patch engine + TUI + sessions + git workflow
```

DeepSeek V4 официально даёт сильную основу: `deepseek-v4-pro`, `deepseek-v4-flash`, 1M context, Thinking/Non-Thinking, OpenAI-compatible API, Anthropic-compatible API, streaming, tool calls, JSON output, context caching и официальные agent-интеграции. Но Anthropic-совместимость неполная: некоторые поля, server tools, image/document/search-result content и MCP-server fields не поддерживаются или игнорируются. Это значит: **для собственного продукта лучше использовать OpenAI-compatible Chat Completions как primary client**, а Anthropic-compatible API держать как adapter для совместимости с чужими agents. ([DeepSeek API Docs][1])

Главная архитектурная ставка:

1. **Rust production core**: быстрый TUI, безопасный shell runtime, deterministic patch engine, хороший installer.
2. **DeepSeek V4 Pro Thinking**: сложная архитектура, debugging, multi-file edits, planning.
3. **DeepSeek V4 Flash Non-Thinking/Thinking**: routing, summaries, file triage, cheap checks, session compaction, reviewer-lite.
4. **Память и индекс обязательны**, даже при 1M context. Long context — это не memory layer.
5. **Patch-first editing**: модель не пишет файлы напрямую; она предлагает patch; patch engine валидирует; пользователь/permission engine решает.
6. **Permission-first UX**: доверие важнее “магии”. Auto modes нужны, но только после audit log, rollback и command classifier.
7. **TUI как основной differentiator**: не просто чат, а terminal workspace: file tree, diff, task plan, tool timeline, command logs, cost meter, session restore.

## DeepSeek-specific product implications

**Primary client:** OpenAI-compatible Chat Completions, потому что там явно видны `thinking`, `reasoning_effort`, `response_format`, `tools`, `stream_options.include_usage`, `reasoning_content`, `tool_calls`, cache hit/miss usage и token accounting. ([DeepSeek API Docs][4])

**Anthropic adapter:** полезен для Claude Code / Anthropic-compatible tools, но нельзя полагаться на Anthropic-native server tools, MCP fields, images/documents/search-result blocks. ([DeepSeek API Docs][9])

**Thinking UX:** технически DeepSeek может возвращать и стримить `reasoning_content`; продуктово лучше показывать пользователю **план, tool timeline и reasoning summary**, а raw reasoning держать в debug/export mode. Это уменьшит шум, токены и риск неправильной интерпретации внутреннего reasoning. ([DeepSeek API Docs][5])

## Differentiators for DSX Code

| Differentiator                                      | Why it matters                                                                   |
| --------------------------------------------------- | -------------------------------------------------------------------------------- |
| **DeepSeek Pro/Flash router**                       | Pro only when needed; Flash for cheap routing/summaries/checks.                  |
| **Cache-aware prompt builder**                      | DeepSeek context caching makes stable prompt prefix economically important.      |
| **Raw reasoning optional, plan visible by default** | DeepSeek can stream reasoning, but UX should show actionable plan/tool timeline. |
| **Patch audit engine**                              | User trust: every edit has provenance, validation, approval, rollback.           |
| **Rust TUI speed**                                  | Fast startup, responsive terminal, easy binary distribution.                     |
| **Local-first memory**                              | Works across sessions without dumping 1M tokens every time.                      |
| **Permission graph**                                | Per-tool, per-path, per-command, per-agent policy.                               |
| **Index + symbol graph**                            | Better than brute-forcing 1M context.                                            |
| **Multi-agent workspace**                           | Useful only after reliable single-agent loop.                                    |

## Where DeepSeek can win

DeepSeek V4’s official pricing and cache model make a Pro/Flash split attractive: Flash is cheap enough for frequent classification, summaries, file triage and small code tasks; Pro can be reserved for heavy reasoning. The 1M context also helps with large monorepos, but only when paired with ranking, summaries and cache-aware prefixing. ([DeepSeek API Docs][3])

## Where DeepSeek can lose

Potential weak spots:

1. **Agent reliability vs Claude/Codex/Gemini**: tool-calling support is confirmed, but real-world coding-agent quality depends on model behavior, not just API features.
2. **Anthropic adapter gaps**: DeepSeek’s Anthropic-compatible API ignores or does not support several Anthropic-native fields, including `mcp_servers`, image/document/search-result blocks, server tool use, web search and code execution blocks. ([DeepSeek API Docs][9])
3. **Dynamic rate limits**: no fixed quota guarantee; product needs queue/backoff/resume. ([DeepSeek API Docs][8])
4. **Thinking token cost/latency**: reasoning can be powerful but expensive/noisy.
5. **Trust**: users will not accept auto shell/file mutations until rollback and permissions are excellent.

# 4. Product vision

**DSX Code** is a **local-first terminal coding workspace** powered by DeepSeek V4. It is not only a CLI and not only a TUI. It is:

```text
CLI command surface
+ interactive TUI
+ local agent runtime
+ project index
+ patch engine
+ permission engine
+ memory store
+ model router
+ MCP/plugin host
```

## Mode 1: Simple CLI mode

Example:

```bash
dsx ask "Why is this test failing?"
dsx edit "Fix the login redirect bug"
dsx plan "Migrate this crate from reqwest blocking to async"
dsx run "Add tests for auth middleware" --no-edit
```

| Field             | Design                                                                                            |
| ----------------- | ------------------------------------------------------------------------------------------------- |
| Target user       | Developer who wants fast one-shot help without opening full TUI.                                  |
| Use cases         | Explain error, inspect files, propose patch, generate commit message, run tests with approval.    |
| Required features | CLI args, project detection, file read/search, patch proposal, diff output, command confirmation. |
| Complexity        | Low to medium.                                                                                    |
| MVP priority      | **Highest**. This becomes the test harness for the core agent.                                    |

## Mode 2: Full TUI mode

Run:

```bash
dsx
dsx tui
```

| Field             | Design                                                                                                   |
| ----------------- | -------------------------------------------------------------------------------------------------------- |
| Target user       | Developer doing real coding tasks in terminal.                                                           |
| Use cases         | Debug loop, multi-file edit, run tests, review diffs, inspect tool calls, compare sessions.              |
| Required features | Chat panel, file tree, diff panel, logs, tool timeline, task plan, permission prompts, model/cost meter. |
| Complexity        | Medium to high.                                                                                          |
| MVP priority      | **High**, but start with minimal layout.                                                                 |

## Mode 3: Agent workspace mode

Run:

```bash
dsx workspace
dsx session list
dsx agent run security-review --background
```

| Field             | Design                                                                                                        |
| ----------------- | ------------------------------------------------------------------------------------------------------------- |
| Target user       | Senior engineers, maintainers, teams, monorepo users.                                                         |
| Use cases         | Parallel subagents, long tasks, background reviews, checkpointed refactors, session dashboard.                |
| Required features | Multi-session manager, subagent runtime, checkpoints, memory, background task scheduler, workspace dashboard. |
| Complexity        | Very high.                                                                                                    |
| MVP priority      | **v1/v2**, not MVP 0.1.                                                                                       |

## B. Chat + coding loop

Core loop:

```text
task -> classify -> collect context -> plan -> ask permissions -> tools
     -> patch proposal -> diff review -> apply -> test -> fix loop -> summary
```

MVP must support:

1. Natural language task.
2. Plan generation.
3. `read_file`, `list_files`, `grep`.
4. Edit proposal as patch.
5. Diff preview.
6. Approve/reject.
7. Command execution with confirmation.
8. Test run.
9. Error analysis.
10. Final summary.

Production adds:

1. Multi-step subagents.
2. Background reviewer.
3. Auto-compaction.
4. Persistent task graph.
5. Semantic search and symbol graph.
6. Checkpoint restore.

# 7. Recommended tech stack

## Stack comparison

| Stack              | Strengths                                                                                                                            | Weaknesses                                                                                  | Best use                                      |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------- | --------------------------------------------- |
| **Rust**           | Fast startup, safe systems code, strong single-binary distribution, excellent async, strong TUI crates, good file/git/index tooling. | Slower initial development, more type/design overhead, async ownership complexity.          | **Production DSX Code**.                      |
| **Go**             | Fast dev, great cross-compile, Bubble Tea/Lip Gloss are polished, simple concurrency.                                                | Less control than Rust for complex patch/index/sandbox internals; less type expressiveness. | Good alternative if team knows Go better.     |
| **TypeScript/Bun** | Fast prototyping, huge plugin ecosystem, easy JSON/MCP integration.                                                                  | Terminal UI and process safety weaker; distribution messier; Node dependency surface.       | Prototype, plugins, optional extension host.  |
| **Python**         | Fastest prototype, Rich/Textual, easy ML tooling.                                                                                    | Performance/distribution/sandbox concerns; less ideal for serious devtool binary.           | Throwaway research prototype or eval harness. |

## Recommendation

**Production:** Rust.

**MVP:** also Rust, but narrow the MVP. Do not try to build the whole TUI/workspace/subagent system at once.

Recommended Rust stack:

| Area           | Crates / approach                                                            |
| -------------- | ---------------------------------------------------------------------------- |
| CLI            | `clap`                                                                       |
| TUI            | `ratatui`, `crossterm`                                                       |
| Async          | `tokio`                                                                      |
| HTTP/SSE       | `reqwest`, `eventsource-stream` or custom SSE parser                         |
| JSON           | `serde`, `serde_json`, `schemars`                                            |
| Config         | `toml`, `figment` or custom merge                                            |
| Git            | Start with `git` CLI; later optional `git2` for speed                        |
| File traversal | `ignore`, `walkdir`                                                          |
| Diff           | `similar`, `diffy`; custom patch apply layer                                 |
| Search         | Start with ripgrep invocation or `grep-searcher`; later `tantivy`/SQLite FTS |
| AST            | `tree-sitter` after MVP                                                      |
| Storage        | SQLite via `sqlx`; JSONL event log for sessions                              |
| Secrets        | OS keychain via `keyring`; env fallback                                      |
| Shell parsing  | `shell-words`, platform-specific handling                                    |
| Tests          | `insta`, `tempfile`, `assert_cmd`, `predicates`                              |
| Logging        | `tracing`, `tracing-subscriber`                                              |
| MCP            | JSON-RPC over stdio/http; implement after core tools                         |

## Can you start in Go/TypeScript and rewrite?

Можно, но с оговорками:

* **Go prototype** acceptable for TUI experimentation.
* **TypeScript prototype** acceptable for prompt/tool orchestration experiments.
* **Full rewrite later** is expensive because agent loop, permissions, patch engine and memory schemas become product core.
* Лучший компромисс: **сразу Rust core**, но можно сделать маленький TypeScript/Go spike для UX inspiration, не как production base.

Where Rust helps:

1. Single-binary distribution.
2. Fast repo scan and diff rendering.
3. Safer command/path handling.
4. Better long-running TUI responsiveness.
5. Strong tests around patch/permissions.
6. Lower memory overhead for large projects.

Where Rust slows you down:

1. Complex UI state modeling.
2. Async streaming + cancellation + borrowed state.
3. Plugin ecosystem.
4. Fast prompt iteration.

Solution: design crates with simple data types; avoid over-abstracting early.

## Routing policy

```text
classify task with Flash non-thinking
  if explanation/small summary:
      Flash non-thinking
  if simple single-file edit:
      Flash non-thinking or Flash thinking
  if unknown risk or multi-file:
      Pro thinking high
  if architecture/refactor/debug/security:
      Pro thinking max
  if reviewing generated patch:
      Flash non-thinking first; Pro fallback on uncertainty
```

## Pseudocode

```rust
async fn run_agent_task(task: UserTask, session_id: SessionId) -> Result<FinalAnswer> {
    let mut session = sessions.load(session_id).await?;
    let policy = permissions.effective_policy(&session.project)?;

    let classification = model_router
        .classify_with_flash_non_thinking(&task, &session)
        .await?;

    let route = model_router.route(RouteInput {
        classification,
        user_mode: session.mode,
        budget: session.budget,
        dirty_git_state: git.status().await?,
    })?;

    let context = context_manager.collect(ContextRequest {
        task: task.clone(),
        route,
        session: session.snapshot(),
        max_tokens: route.context_budget(),
        include_git_diff: true,
        include_memories: true,
        include_file_summaries: true,
    }).await?;

    let mut plan = planner.generate_plan(&task, &context, route).await?;
    session.record(Event::PlanCreated(plan.clone())).await?;

    if session.mode == Mode::PlanOnly {
        return Ok(FinalAnswer::from_plan(plan));
    }

    for step_idx in 0..MAX_AGENT_STEPS {
        let prompt = prompt_builder.build_agent_prompt(PromptInput {
            task: task.clone(),
            plan: plan.clone(),
            context: context_manager.refresh_for_step(&session).await?,
            tool_specs: tool_registry.specs_for_policy(&policy),
            mode: session.mode,
        })?;

        let mut stream = deepseek.stream_chat(prompt, route).await?;

        while let Some(event) = stream.next().await {
            match event? {
                ModelStreamEvent::ReasoningDelta(delta) => {
                    // UI can show summary/debug stream, not required for core logic.
                    session.record(Event::ReasoningDelta(delta.redacted())).await?;
                }

                ModelStreamEvent::ContentDelta(delta) => {
                    ui.emit(UiEvent::AssistantText(delta)).await?;
                }

                ModelStreamEvent::ToolCall(call) => {
                    let validated = tool_registry.validate(call)?;

                    let decision = permission_engine.decide(PermissionRequest {
                        tool: validated.name.clone(),
                        args: validated.args.clone(),
                        mode: session.mode,
                        risk: risk_classifier.score(&validated),
                        cwd: session.project.root.clone(),
                    })?;

                    match decision {
                        PermissionDecision::Allow => {
                            let result = tool_runtime.execute(validated).await?;
                            session.record(Event::ToolResult(result.clone())).await?;
                        }

                        PermissionDecision::Ask(request) => {
                            let approved = ui.ask_approval(request).await?;
                            if approved {
                                let result = tool_runtime.execute(validated).await?;
                                session.record(Event::ToolResult(result.clone())).await?;
                            } else {
                                session.record(Event::ToolDenied(validated.id)).await?;
                            }
                        }

                        PermissionDecision::Deny(reason) => {
                            session.record(Event::ToolDeniedWithReason {
                                tool_call_id: validated.id,
                                reason,
                            }).await?;
                        }
                    }
                }

                ModelStreamEvent::Usage(usage) => {
                    session.usage.add(usage);
                    ui.emit(UiEvent::UsageUpdated(session.usage.clone())).await?;
                }

                ModelStreamEvent::Done => break,
            }
        }

        if patch_engine.has_pending_patch(&session).await? {
            let patch = patch_engine.pending_patch(&session).await?;
            patch_engine.validate(&patch).await?;
            ui.show_diff(&patch).await?;

            match permission_engine.patch_decision(&patch, session.mode).await? {
                PatchDecision::Apply => {
                    git.maybe_checkpoint("before dsx patch").await?;
                    patch_engine.apply(&patch).await?;
                    session.record(Event::PatchApplied(patch.id)).await?;
                }
                PatchDecision::Reject(reason) => {
                    memory.record_rejected_patch(&patch, reason).await?;
                    session.record(Event::PatchRejected(patch.id)).await?;
                }
                PatchDecision::Revise(feedback) => {
                    session.record(Event::PatchRevisionRequested(feedback)).await?;
                    continue;
                }
            }
        }

        if test_policy.should_run_tests(&session, &plan) {
            let cmd = test_policy.select_test_command(&session.project)?;
            let result = command_runner.run_with_permission(cmd, &policy).await?;
            session.record(Event::CommandResult(result.clone())).await?;

            if result.failed() {
                plan = planner.update_after_failure(plan, result).await?;
                continue;
            }
        }

        if plan.is_complete(&session) {
            break;
        }
    }

    let final_answer = summarizer.finalize(&task, &session).await?;
    memory.extract_and_store(&session, &final_answer).await?;
    sessions.save(session).await?;

    Ok(final_answer)
}
```

# 11. Memory and context system

## Principle

**1M context is a capacity feature, not a memory system.** Dumping everything into context is expensive, slow, and unreliable. DSX needs a persistent retrieval layer.

## Memory types

| Memory type           | Store                   | Update trigger                     | Prompt usage                               | Forget/archive policy                     | Pollution prevention                       |
| --------------------- | ----------------------- | ---------------------------------- | ------------------------------------------ | ----------------------------------------- | ------------------------------------------ |
| Project index         | SQLite + index files    | Repo scan, file change             | Select relevant files/symbols              | Rebuild on file hash change               | Never store ignored/vendor unless allowed. |
| File summaries        | SQLite                  | After reading/editing file         | Summarize large files instead of full text | Invalidate on file hash change            | Store summary with hash and source lines.  |
| Session history       | JSONL + SQLite metadata | Every turn/tool/patch              | Resume and summarize                       | Compact old sessions                      | Keep raw logs local, redact secrets.       |
| Decision log          | SQLite                  | User approves/rejects plan/patch   | Respect prior architecture choices         | Archive after project config change       | Require source event IDs.                  |
| Command log           | SQLite                  | Command execution                  | Suggest test/build commands                | Truncate output; keep structured result   | Redact secrets; store exit code.           |
| Error memory          | SQLite                  | Failed tests/builds                | Recognize recurring failures               | Expire after dependency changes           | Must link to command result.               |
| Tool result log       | JSONL                   | Every tool call                    | Debug/audit                                | Compact old results                       | Store checksums for large outputs.         |
| Patch history         | SQLite                  | Patch proposed/applied/rejected    | Avoid repeated bad edits                   | Keep applied; archive rejected after time | Include reason and validation result.      |
| Rejected patch memory | SQLite                  | User rejection                     | Avoid re-proposing                         | Expire unless user marks permanent        | Store specific reason, not vague “bad”.    |
| User preferences      | Global SQLite/TOML      | Explicit user setting              | Style, verbosity, preferred commands       | User editable                             | Do not infer sensitive preferences.        |
| Task checkpoints      | Git + SQLite            | Before patch, before risky command | Rollback/restore                           | Keep recent N                             | Include dirty state hash.                  |
| Dependency map        | SQLite                  | Package file change                | Context and command selection              | Rebuild on lockfile/package changes       | Do not infer uninstalled deps.             |
| Symbol graph          | SQLite/Tantivy          | Index build                        | Semantic/symbol search                     | Rebuild per file hash                     | Tree-sitter confidence flag.               |

# 12. Tool execution and safety

## Command classifier

Risk score inputs:

```text
risk = command pattern
     + cwd/path scope
     + writes filesystem?
     + network?
     + package install?
     + git destructive?
     + secret access?
     + sudo/admin?
     + output sensitivity?
     + user permission mode
```

## Risk levels

| Level     | Examples                                                                                     | Default action              |
| --------- | -------------------------------------------------------------------------------------------- | --------------------------- |
| `read`    | `pwd`, `ls`, `git status`, `git diff`, safe grep/search                                      | Allow in trusted workspace  |
| `low`     | language version checks, formatting check, non-writing lint                                  | Allow or ask depending mode |
| `medium`  | tests/builds that write artifacts, formatter that modifies files                             | Ask                         |
| `high`    | dependency install, network calls, migrations, git commit/reset, deleting files              | Ask with strong warning     |
| `blocked` | destructive outside workspace, credential exfiltration, privilege escalation, broad deletion | Deny by default             |

## Commands allowed without confirmation in default `ask` mode

Only if cwd is inside workspace and args pass validation:

```text
pwd
git status
git diff
git log --oneline ...
git branch --show-current
rg / grep search
find/list safe variants
cat/sed/head/tail for workspace files
language version commands: rustc --version, cargo --version, node --version
package manager metadata read commands where non-mutating
```

## Commands requiring confirmation

```text
cargo test / npm test / pnpm test / pytest / go test
cargo build / npm run build
formatters that write: cargo fmt, prettier --write, rustfmt
package installs/adds/removes
database migrations
network fetches
git commit
git checkout branch
git reset / git clean
opening browser/web fetch
running project scripts
```

## Commands denied by default

Patterns:

```text
privilege escalation: sudo, su, doas
destructive broad deletion outside workspace
changing ownership/permissions broadly
disk/device manipulation
credential/token file exfiltration
curl|wget piped into shell
commands targeting parent directories with destructive actions
git clean -fdx without explicit user override
git reset --hard without explicit checkpoint and approval
```

## Sandbox design

MVP:

1. Workspace cwd restriction.
2. Timeout.
3. Output truncation.
4. Env redaction.
5. Command risk classifier.
6. Confirmation prompts.
7. Git checkpoint before edit.

Production:

1. macOS sandbox profile where feasible.
2. Linux bubblewrap/firejail optional.
3. Windows/WSL policy wrapper.
4. Network deny/allow by domain.
5. Per-command filesystem allowlist.
6. Per-MCP server permission policy.
7. Secret scanning before sending model context.

## Output truncation

```text
max_stdout_bytes: 64 KB default
max_stderr_bytes: 64 KB default
keep_head: 16 KB
keep_tail: 48 KB
parse_known formats: cargo, pytest, npm, go test
```

## Secret redaction

Redact before logs/model:

```text
API keys
JWT-like tokens
SSH/private keys
.env values
Authorization headers
cloud credentials
database URLs with passwords
```

# 13. File editing / patch engine

## Edit pipeline

```text
1. read_file
2. generate patch
3. validate patch
4. show diff
5. approve/reject/revise
6. apply patch
7. format/lint if approved
8. run tests
9. rollback if failed and user chooses
10. audit log
```

## Validation rules

1. Path must be inside workspace.
2. File must not be ignored unless explicitly allowed.
3. Patch must apply to current file hash.
4. Hunk context must match.
5. No binary edits unless explicit tool.
6. No secret file edits unless explicit approval.
7. Multi-file patch has atomic apply or clean rollback.
8. Large patch requires extra confirmation.
9. Generated file/vendor changes require warning.
10. Rejected patch is stored to avoid repeating.

# 16. Installation and distribution

## Distribution channels

| Platform    | MVP                                                | Production                                              |
| ----------- | -------------------------------------------------- | ------------------------------------------------------- |
| macOS       | Homebrew tap + GitHub release binary               | Signed/notarized binary, Homebrew core if possible      |
| Linux       | install script + release tarball + `cargo install` | deb/rpm/AppImage/Nix                                    |
| Windows     | `cargo install`, release exe                       | winget/scoop/choco, Windows Terminal support            |
| WSL         | Linux binary                                       | First-class recommended Windows path                    |
| npm wrapper | Optional                                           | Useful for `npm i -g dsx-code`, downloads native binary |
| Self-update | Not MVP                                            | Signed release update check                             |

## API key storage

Priority order:

1. OS keychain.
2. `DEEPSEEK_API_KEY` env var.
3. Explicit config reference to env var.
4. Never store raw key in project config.

First-run flow:

```text
dsx
 ├─ detect project
 ├─ ask for API key or env var
 ├─ store in keychain
 ├─ verify key with tiny model request or models endpoint
 ├─ create .deepseek-code/instructions.md
 ├─ scan project
 └─ open TUI
```

DeepSeek official docs expose model/API endpoints and model names; validation should use a low-cost request and never print the key. ([DeepSeek API Docs][1])

## Update system

MVP:

```bash
dsx --version
dsx doctor
```

Production:

```bash
dsx update
dsx update --check
```

Update safety:

1. Verify release signature/checksum.
2. Do not auto-update during active session.
3. Keep previous binary for rollback.
4. Show changelog.

# 17. MVP / v1 / v2 roadmap

## MVP 0.1: reliable single-agent core

| Area    | Features                                                          |
| ------- | ----------------------------------------------------------------- |
| Model   | DeepSeek client, Pro/Flash routing, streaming, usage/cost.        |
| CLI     | `dsx ask`, `dsx edit`, `dsx plan`, `dsx tui`.                     |
| Project | Scan repo, respect gitignore, file tree.                          |
| Tools   | read/list/grep/git status/git diff/run command with confirmation. |
| Patch   | Propose patch, validate, diff review, apply/reject.               |
| Safety  | ask mode, read-only, plan-only, dry-run.                          |
| Session | Save/resume basic session JSONL.                                  |
| Config  | global + project TOML.                                            |

Acceptance criteria:

1. Can fix a simple bug in a Rust/JS project.
2. Never edits without approval in ask mode.
3. Failed patch never corrupts file.
4. Session can resume after interrupted stream.
5. Cost/token stats displayed.
6. 50 benchmark tasks run with logs.

## MVP 0.2: memory + better TUI

| Area    | Features                                                  |
| ------- | --------------------------------------------------------- |
| Memory  | SQLite memory, file summaries, command log, decision log. |
| Git     | Checkpoints, rollback, branch helper, commit message.     |
| TUI     | Better panels, command palette, session list.             |
| Tests   | Test command detection, failure parsing.                  |
| Safety  | More risk rules, secret redaction.                        |
| Routing | Better classifier, cost policy.                           |

Acceptance criteria:

1. Can restore prior session summary.
2. Can avoid repeating rejected patch.
3. Can parse common test failures.
4. Can rollback patch via checkpoint.

## v1: extensible agent platform

| Area        | Features                                                       |
| ----------- | -------------------------------------------------------------- |
| Subagents   | Planner, scanner, implementer, reviewer, test-fixer, security. |
| MCP         | MCP servers and permission model.                              |
| Index       | Tree-sitter symbols, semantic/symbol search.                   |
| Plugins     | Custom tools/prompts/hooks.                                    |
| Permissions | safe-yolo, allow/deny by path/tool/command.                    |
| Team        | team config, shared instructions.                              |

Acceptance criteria:

1. MCP tool cannot bypass permissions.
2. Symbol search improves context selection benchmark.
3. Subagents improve success rate without runaway cost.
4. Team config precedence works.

## v2: workspace and enterprise

| Area              | Features                                 |
| ----------------- | ---------------------------------------- |
| Background agents | Long-running tasks, queue, dashboard.    |
| Multi-session     | Workspace dashboard, parallel sessions.  |
| IDE bridge        | VS Code/Zed/JetBrains bridge.            |
| Remote agents     | Optional remote sandbox workers.         |
| Cloud sync        | Optional encrypted sync.                 |
| Enterprise        | Policies, audit export, managed configs. |

Acceptance criteria:

1. Background task can pause/resume safely.
2. Remote execution cannot access disallowed paths/secrets.
3. Team policy cannot be weakened by project config.

# 20. Risks and trade-offs

| Risk                                                    | Severity | Probability | Mitigation                                                                     | MVP impact            |
| ------------------------------------------------------- | -------: | ----------: | ------------------------------------------------------------------------------ | --------------------- |
| DeepSeek tool calling works but model produces bad args |     High |      Medium | Strict schemas where possible; validate all args; reject unknown paths.        | Must solve in 0.1.    |
| DeepSeek API changes/pricing changes                    |   Medium |      Medium | Provider abstraction; pricing config fetched/updated manually; versioned docs. | Moderate.             |
| Dynamic rate limits / 429                               |   Medium |      Medium | Retry/backoff, queue, resumable streams.                                       | Must handle.          |
| 1M context abuse                                        |     High |        High | Context manager, budgets, summaries, index, cache-aware prefix.                | Must avoid.           |
| Latency too high with Pro Thinking max                  |   Medium |      Medium | Route only hard tasks to max; Flash for triage.                                | Must route.           |
| Cost too high                                           |   Medium |      Medium | Cost meter, budget caps, Flash summaries, cache prefix.                        | Must show usage.      |
| Hallucinated patches                                    |     High |        High | Patch validation, read-before-edit rule, diff review.                          | Core MVP.             |
| Unsafe shell commands                                   | Critical |      Medium | Risk classifier, denylist, sandbox, confirmations.                             | Core MVP.             |
| TUI complexity delays product                           |   Medium |        High | Build CLI core first, minimal TUI second.                                      | High schedule risk.   |
| Rust slows iteration                                    |   Medium |      Medium | Keep crates small, use fixtures, avoid overengineering.                        | Manageable.           |
| Windows compatibility                                   |   Medium |      Medium | WSL first, Windows later; path abstractions.                                   | Defer polish.         |
| MCP complexity/security                                 |     High |      Medium | v1 only; same permission engine; tool schema validation.                       | Not MVP.              |
| Memory pollution                                        |   Medium |        High | Source IDs, confidence, expiry, explicit extraction prompt.                    | Start simple.         |
| Subagent overhead                                       |   Medium |        High | Make subagents prompt roles first; measure cost.                               | v1.                   |
| User trust problem                                      | Critical |      Medium | Transparent diff/tool timeline/audit/rollback.                                 | Product-defining.     |
| Diffs too large                                         |   Medium |        High | Patch size thresholds, chunked review, ask for scope.                          | MVP guard.            |
| Command output too long                                 |   Medium |        High | Truncate, parse, summarize, store raw local only.                              | MVP.                  |
| Secrets leakage to model/logs                           | Critical |      Medium | Redaction, ignore rules, secret scanner, deny `.env`.                          | Core MVP.             |
| Anthropic API compatibility gaps                        |   Medium |      Medium | Native OpenAI-compatible client primary; Anthropic adapter secondary.          | Architectural choice. |
| Raw reasoning UX confusion                              |   Medium |      Medium | Show plan/tool timeline by default; raw debug only.                            | UX choice.            |

3. Full MCP marketplace.
4. IDE extensions.

These will distract from the real hard problem: **safe, reliable, reviewable local code changes**.

# 23. Final recommendation

Build **DSX Code** as a Rust local-first coding-agent runtime with DeepSeek V4 as the model provider.

The MVP should not try to match every feature of Claude Code, Codex CLI, Gemini CLI, OpenCode, Cursor and Zed. It should beat them in one narrow area first:

```text
A beautiful terminal agent that can inspect a repo,
propose correct patches,
show clean diffs,
run approved commands,
track cost/tokens,
save sessions,
and never silently mutate the project.
```

Use this product architecture:

```text
Rust CLI/TUI
  -> deterministic agent loop
  -> DeepSeek Pro/Flash router
  -> permission engine
  -> patch engine
  -> local memory/index
  -> git checkpoints
  -> MCP/plugins later
```

The most important design choice: **DeepSeek V4 is not the agent. DSX is the agent.** DeepSeek supplies reasoning, code generation and tool-call intent; DSX supplies trust, context, tools, permissions, memory, patches, rollback and UX.

Start with:

```bash
dsx plan "..."
dsx edit "..."
dsx
```

Ship MVP 0.1 when it can safely complete real small coding tasks with approval and rollback. Then add memory, git workflow, subagents, MCP, workspace mode and IDE bridge.
